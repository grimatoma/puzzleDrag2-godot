Visual Golden Audit

A screenshot-by-screenshot review of all 149 visual-regression goldens across the two codebases in this repo — the Godot 4.6 port and the React+Phaser app — answering: is each golden showing what we expect, or is it trash capturing a bug? Plus why the Godot suite has 36 goldens vs Phaser's 113, and which goldens to add.

Audit date 2026-06-10 · Goldens frozen: Godot 2026-06-08, Phaser current · Method: 32 image-review agents + 14 adversarial verifiers, reconciled against current source

149 goldens reviewed Godot: 36 Phaser: 113 2 real bugs 6 trash goldens 36 stale

Overview

The repo has two separate visual-regression golden sets: the React+Phaser app (tests/visual/__goldens__, 113 PNGs) and the Godot 4.6 port (godot/tests/visual/__goldens__/Windows, 36 PNGs). I reviewed every one against what its scenario claims it should show, then reconciled each finding against the current source to separate "the image is stale" from "the code has a bug."

TL;DR. The surprise is where the problems live. The Godot port's code is fine — but its entire 36-golden set is stale (frozen 06-08; every screen was restyled 06-10 and Windows goldens are never pixel-diffed in CI, so the rot is invisible). The mature Phaser suite is where the genuine defects are: 2 real product bugs (a raw tile_grass_hay key leaking into player copy), ~6 trash goldens that pass CI while verifying nothing (empty/duplicate captures), and 1 scenario that under-demonstrates its content. The "36 vs 113" gap is half illusory (Phaser counts two extra apps + desktop re-shoots) and half real (the port's board has 2 goldens where Phaser has 17, despite the port having mine/fish/tools/bosses).

How this was done

  • Fan-out review. 32 agents each opened a batch of golden PNGs (Godot batches were given the corresponding Phaser reference to compare port fidelity) and returned a structured per-golden verdict describing the actual visible content.
  • Adversarial verify. Every golden flagged as bug/trash/uncertain (14) was independently re-opened by a second, skeptical agent. This caught 4 false positives — goldens the first reviewer wrongly flagged that are actually correct.
  • Source reconciliation (the key step the agents can't do). An image only shows the rendered pixels — it can't tell you whether they match today's code. I traced every flagged defect to current source. This is what revealed the Godot set is wholesale stale: the visible "bugs" (raw keys, mis-categorised tiles) are already fixed in current GDScript; the goldens just predate the fix.
Note — one workflow artifact. The verify step had an id collision: story-prompt exists in both suites, and the verifier re-read the Godot golden (which renders fine) when clearing the Phaser flag. I opened the Phaser story-prompt golden by hand — it is a blank town (trash), part of the dialogs-disabled cluster below. Findings here reflect the hand-corrected truth.

Godot port — the whole set is stale

In the image review, the 36 Godot goldens mostly looked fine as pictures — real screens, real tile art, readable labels. But an image review can't see that they no longer match the code. They don't.

Root cause. Every Godot golden was last refreshed 2026-06-08. On 2026-06-10 a sweep of UI work landed that restyled essentially every screen — most decisively commit 644fc1eb "Unify menu surfaces: depth-styled backdrops/scrims, carded Town ledger, status-line alignment", which touched 26 scene scripts (the screen behind every single golden), plus 81a158da (modal-card-surface dedupe) and f195d45d (06-09, TileCollectionScreen rewrite). These are static visual changes, not just animation, so they move settled pixels beyond the harness's 1% tolerance.
Why CI never caught it. run_visual_tests.gd only pixel-diffs when run non-headless on the golden platform (Windows). CI runs on Linux under xvfb, where there is no committed Linux golden, so every scenario falls back to a render-smoke (size + non-blank) check. The Windows goldens are therefore never diffed in CI — they rot silently until someone runs the diff locally. That someone is now overdue.

The image review flagged several raw-key / mis-categorisation defects on the Godot goldens. I traced each to current source — all are fixed; the goldens simply predate the fix. This is the signature of staleness, not live bugs.

Two current items worth a glance (low priority)

Desktop stockpile copy minor

The desktop board goldens (board-farm-idle/desktop, board-farm-chain/desktop) show a stockpile header reading "9 kinds" while the body says "Stockpile empty" — a copy contradiction. Worth confirming against current Hud.gd when re-baselining.

Desktop framing: stretch vs pillarbox minor

The harness header comments describe the desktop viewport as pillarboxing the 720×1280 portrait content, but the goldens show a responsive full-window stretch (the project uses stretch/aspect="expand"). Not a bug — but the harness doc and the actual behaviour disagree; align one to the other.

One reviewer false-positive was also confirmed: story-prompt (Godot) correct — it renders the "A Cold Hearth / WELCOME TO HEARTHWOOD VALE" modal with Skip/Next over a dimmed board, exactly as expected.

Phaser app — where the real defects are

The mature React+Phaser suite is generally healthy (≈100 of 113 goldens are correct), but it holds every genuine problem this audit found: live product bugs the goldens have been updated to accept, and a cluster of goldens that pass CI while capturing the wrong screen.

A · Real product bugs (fix the code, not the golden)

Raw tile key leaks into player-facing copy. The bonus/ability summary renders "Boosts tile_grass_hay by 1" instead of "Boosts Hay by 1". Visible in goldens start-farming-tile-chooser and tiles-activate-grass-spiky (and tiles-research-progress).

Root cause — src/ui/primitives/BrowserDetail.tsx:205:

case "pool_weight":
case "pool_weight_legacy":
  rows.push(`Boosts ${p.target ?? "spawn"} by ${p.amount ?? 1}`);   // p.target = "tile_grass_hay"
  break;
case "threshold_reduce":
  rows.push(`Reduces ${p.target ?? "a chain"} by ${p.amount ?? 1}`); // same pattern — also leaks
case "recipe_input_reduce":
  rows.push(`Reduces ${p.recipe} ${p.input} by ${p.amount ?? 1}`);   // and here

p.target / p.recipe / p.input are raw catalog keys, interpolated with no mapping through ITEMS[…].label or pretty_name. The fix is one resolver applied to those interpolations. The goldens were updated to accept this output, so CI is green on a live bug — a reminder that test:visual:update can launder a regression into the baseline.

Irony worth noting: the Godot port already solved exactly this class of bug (every Godot screen routes keys through UiKit.pretty_name()). Here the React original is the one leaking.

B · Trash goldens — green in CI, but verify nothing

These goldens pass on every PR (the render reliably reproduces the same wrong screen, so it matches the committed baseline) yet capture none of the content their scenario name promises — a false sense of coverage.

B1 · Dialogs-disabled cluster

The harness disables the very thing these scenarios test. Every Phaser visual spec sets window.__HEARTH_DISABLE_DIALOGS__ = true (to suppress auto-popping story/season modals). But src/ui/Modals.tsx gates the story-beat modal (line 512), season modal (38), and toast bubble (613) behind isDialogsDisabled(). So any scenario whose payload is one of those renders an empty town.

B2 · Dead-route cluster

Scenarios navigate to routes that no longer exist. The matrix sends townsfolk-castle → #/townsfolk/castle and townsfolk-bosses → #/townsfolk/bosses, but src/features/townsfolk/index.tsx:9 defines TABS = ["workers", "quests"] and falls back to "workers" for anything else. Both scenarios silently render the Workers tab — and are byte-identical to each other and to townsfolk-workers.

Castle and the boss gallery moved out of the townsfolk tabs. Fix: retarget these scenarios to wherever castle / bosses now live (or drop them).

C · Scenario coverage shortfall

board-farm-fire-rats under-demonstrates

A board-wide warm fire/rats atmosphere tint renders (≈60% of pixels differ from idle), but there are no fire tiles and no rat sprites on the board. The scenario's state builder (stateBuilders.ts:470) sets hazards.fire.cells/hazards.rats but ships a grid with no fire/rat cells, and the visual harness applies the grid via a path (rebuildGridFromState) that never paints the fire overlays (those only happen inside GameScene.fillBoard). So the golden technically matches its scenario but fails to show its headline content. Fix the scenario to inject fire/rat cells (or route through fillBoard).

D · Reviewer over-flags (verified actually correct)

The adversarial verify pass earned its keep — it overturned 4 first-pass flags:

  • inventory-grid-all (desktop) correct — it is a 4-column grid of cards; the reviewer mistook it for a list.
  • balance-recipes-category & balance-ability-article (Dev Panel, desktop) correct — fully rendered; the claimed "Loading…" placeholder is not present.
  • story-prompt (Godot) correct — renders the modal (the verifier read the Godot copy due to the id collision noted in Overview).

E · Minor / cosmetic

  • map-keeper-choice — emoji glyphs render as tofu boxes (□) under headless Chromium's missing emoji font. Known, benign, suite-wide.
  • town-build-picker-ready — the picker's focused detail is an already-built Hearth (disabled Build) rather than an enabled, buildable option; the enabled rows are in the list but not in focus.
  • town-market-news — no obvious market-news ticker/banner is visible; verify the scenario actually triggers the ticker.
  • board-boss-minimized / board-boss-active — the boss strip/banner isn't clearly distinguishable between the two; flagged low and not independently verified — worth a manual look.

Why 36 vs 113?

Half the gap is an apples-to-oranges artifact; half is a real board-coverage hole.

  • 19 goldens can't apply to the port — the Dev Panel (/b/, 13) and Story Tree Editor (/story/, 6) are separate Vite apps / authoring tools; the Godot port is the game only.
  • 20 are desktop re-shoots of mobile scenarios (Phaser's desktop-smoke set is larger; the port shoots only 6 representative desktop framings).
  • The real comparison is 80 Phaser game (mobile) vs 30 Godot game (portrait) — and most of that ~50 difference is many Phaser variants collapsed to one Godot golden (inventory 5→1, tiles 8→1, map 7→2, townsfolk 4→2, quests 2→1, achievements 2→1, crafting 2→1) plus the board hole below.

The one real hole: the board

The core game surface is the least-covered. The Godot board has 2 goldens (board-farm-idle, board-farm-chain). The Phaser board has 17. Yet the parity matrix marks the port's mine, fish, tools, bosses, and hazards as ported (full/partial) — so these goldens can be captured now; they just haven't been.

"Each golden test should describe what it should be showing"

Phaser weak

Each scenario already carries an expectation string (matrix.ts → buildExpectationForScenario), printed to the test log and pushed as a test annotation. But only 4 are hand-written; the rest are auto-generated boilerplate ("Scenario X renders expected #/y UI state") that says nothing reviewable. Recommend: write a real one-line expectation per scenario (especially board/tool/boss/story), like the existing overrides.

Godot missing

run_visual_tests.gd scenario rows are just {id, deeplink, seed} — "what it should show" lives only in code comments on the seed functions. Recommend: add an expect: field to each row and print it in the "N checks" output, so every Godot golden self-documents its intent (mirroring Phaser's expectation). This is the single cheapest durable improvement.

Golden-regen gotcha (cost us hours — worth recording). playwright.visual.config.js uses reuseExistingServer: true on port 5173. If any other dev server is on 5173 (e.g. the main repo's, or a stale one), the visual run silently tests that codebase, producing goldens that don't match this worktree. Regenerate with no other 5173 server up — or temporarily point the config at a free port with reuseExistingServer:false + --strictPort. (Reading the same golden path twice in one session can also return a cached image — copy to a fresh path to re-inspect.)
