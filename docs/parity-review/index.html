Hearthlands — Godot Port Gap Review

A screen-by-screen, feature-by-feature comparison of the Godot 4 port against the live React+Phaser game, focused on UI and functionality. Every screen below was rendered live from both builds and placed side by side — the matrix was not trusted.

Reviewed 2026-06-08 · mobile-portrait (React 390×844 / Godot 720×1280) · React = reference (left, green), Godot = port (right, ember)

core loop solid 6 major UI gaps long-tail screens ≈ parity 37 fix tasks

Verdict

The Godot port is far from "barebones" — it is a mature, playable reimplementation. The full Town 1→2→3 vertical slice runs end-to-end, the chain/collapse/refill mechanic is faithful, the mine and harbor boards render their tile art well, and the long-tail screens (achievements, quests, chronicle, charter, castle, daily, decorations, portal) are near pixel-parity with React. The logic backbone is healthy (headless suites green).

TL;DR. The gaps are concentrated in a handful of high-traffic surfaces: the town map building art, the Craft screen, the tile collection, the board stockpile / season strip, the Townsfolk "Workers" tab, and the end-of-run summary. Fixing those six closes most of the perceptual distance between the two builds. Everything else is either small polish or a deliberately-parked system (boons, multi-settlement world, weather).

How to read the montages. In every comparison image the left half is React (the reference) and the right half is the Godot port. Both are live renders captured during this review.

Puzzle board P1

  • Chain mechanic, 8-way adjacency, collapse/refill and animated juice are all present and feel right.
  • Stockpile placement & prominence. React shows a rich STOCKPILE panel near the top (below the tools, above the board) with a grid of resource chips and "6/12 KINDS". Godot relegates it to a thin one-line strip at the very bottom that reads "Stockpile · 0 kinds / empty" for most of a run — the player can't see what they're gathering at a glance.
  • Season strip is under-built. React's board header is a prominent 4-season strip (spring/summer/autumn/winter icons) with the current season highlighted and a bold "8 TURNS LEFT" pill. Godot shows a much thinner season bar and leans on the top-bar town pills instead.
  • Tile variety & density is thinner on the Godot farm board (more grass/tree-heavy, slightly smaller cells with wider cream gutters). React's board reads more colourful and full-bleed.
  • Top bar differs in intent: React's board chrome is gameplay-first (season + turns); Godot keeps the persistent town top-bar (town name + coins + tier pill) on the board too.

Mine & Harbor boards parity

  • Mine tiles (grey ore, black coal, blue gems) render distinctly and match React closely.
  • Harbor tiles (fish, shells, sand, giant pearl) render with the blue tide tint; the "high/low tide" indicator is present.
  • Same board-page caveats as the farm board apply (stockpile at bottom, thinner season strip).

Town map P1

  • The spatial layout itself is a faithful port — same lot grid, river, bridges, lampposts, central well, scattered trees, pan/zoom controls, and the "Build · N/M plots" CTA.
  • Buildings have no distinct art. React renders each building type as a unique illustrated structure — a windmill, a red barn (the farm field), a forge/smithy, a glowing purple portal, and several varied cottages/manors/silos. Godot draws every built plot as the same generic tan box with a small red roof, disambiguated only by a tiny text label ("lumber_camp", "coop", "garden", "bakery", "kitchen"). This is the single biggest "looks unfinished" signal in the port.
  • No townsfolk. React animates little villager figures walking the streets between plots; the Godot town is empty.
  • React's town header carries a "✨ Boons" button and a zone-spawn "i" info pill ("Zone spawn rates and modifiers"); Godot has neither (boons is dropped; zone info is unsurfaced).
Highest-impact fix. Porting the React iso-building set (src/iso/, 27 procedural SVG buildings) so each building_id maps to its own sprite would, on its own, move the port's perceived polish more than any other single change.

Craft screen P1

  • Wrong screen, essentially. React's "Craft" tab is a dedicated crafting interface: station tabs (Bakery / Larder / Forge / Workshop), a recipe grid (Bread Loaf, Honey Roll, Harvest Pie, Festival Loaf, Wedding Pie…), and a rich detail card with the recipe description, input chips ("Flour 12/2, Eggs 12/1"), a worker-bonus line, and a green Craft button.
  • Godot's "Craft" tab is instead a town-management ledger — Settlement info, a Buildings list with Demolish buttons, a 2-row Refine section, and a long Market sell/buy list. Useful, but it is not the crafting experience and mixes four concerns into one scroll.
  • Only 2 recipes (Bread, Supplies) versus React's multi-station catalog. No recipe descriptions, no recipe cards, no station tabs.
  • Market buy does exist (a "— Buy —" section below the sell fold) — it's just buried.

Inventory P2

  • Godot's inventory is arguably richer in places: it groups items by category (Farm Goods / Refined / Mine), shows per-item sell value ("12🪙 ea · 480🪙"), and has a working search field.
  • Missing the "Items" tab (React has All / Resources / Tools / Items; Godot has All / Resources / Tools).
  • Missing the grid/list view toggle (the ⊞ button in React's header).
  • React item icons are larger and framed; Godot's are smaller. Minor.

Start-farming & lifecycle parity

  • Faithful port: the 6-tile-type chooser, "Turns this run: 10", "Cost to start: 50🪙", and Start/Cancel all match.
  • Entry point differs slightly: React taps the farm-field building on the map ("Enter Farm Field"); Godot taps a farm board-pad / "▶ Board" button. Both open the same modal.

End-of-run summary P1

  • React closes a run with a rich recap: chains played, longest chain, "best moment" callout, what-you-brought-home item tallies, upgrades crafted, coins gained, which NPCs noticed you (bond deltas), story beats fired, and supplies consumed.
  • Godot shows a single-line modal ("A full year of 10 turns is in. Your stores: 184 coins. +25🪙 return bonus. Return to Town.") — correct, but it throws away the satisfying payoff screen.
  • The lifecycle plumbing is sound (the harvest modal fires at the right time and routes back to town); only the content is thin.

Tile collection P1

  • Godot lists all 79 tiles grouped by category with art, name and produced resource — honest and complete coverage.
  • No category tabs (React: Farm / Mining / Water / Hazards / Uncategorized) and no sub-category tabs (Grass / Grain / Bird / Veg / Fruit / Flower / Trees).
  • No per-tile detail panel — React shows tier, a flavour description, and a "Bonuses" block per tile.
  • No tile-ability "Activate" system. React tiles carry abilities (e.g. Spiky Grass, Turkey → "2 free moves per season") that the player activates from this screen. Godot's screen explicitly notes the discovery/ability economy is not ported.

Townsfolk & Workers P1

  • The "Workers" tab shows the wrong thing. In React, "Workers" is the worker-hiring system — hire a Farmer / Lumberjack / Miner / Baker, each with a card showing next-hire cost, current effect ("−2 grain chain steps"), and bonuses, plus Hire/Fire buttons. In Godot, the "Workers" tab shows the 5-NPC bond roster (Wren/Scout, Mira/Baker, … with "bond 5.0/10 ×1.00 orders").
  • The worker-hire system does exist in the port, but it's buried in the Craft/Town list, not on this tab — so the tab label and content are mismatched versus React.
  • The NPC bond roster itself is nicely built (coloured avatars, role, tier-tinted bond bars) — it just needs its own tab/home.
  • React's Townsfolk hub also folds in Bosses and Castle sub-tabs; Godot reaches those as separate menu screens.

Quests & Almanac parity

  • Both have Quests/Almanac tabs, progress bars, coin + XP rewards, and gated Claim buttons. Functionally equivalent.
  • React's quest cards are larger and carry an item icon + a bigger green Claim; Godot's rows are more compact and icon-less. Cosmetic.

Achievements parity

  • Strong parity: Trophies/Collection tabs, 2-column card grid, category groups, progress bars, unlocked/locked states.
  • Godot exposes 19 port-reachable trophies vs React's 27 (the extras depend on unported systems). Expected.

Chronicle · Charter · Castle parity

  • Chronicle: the Act-grouped timeline rail with beat cards + quotes is a faithful port.
  • Charter: the six Hollow-Pact terms with honored/pending/violated pills and the Terms/All-choices toggle match React closely.
  • Castle: the per-need contribution bars with Contribute 1 / All(N) are present and correct.

Portal & Decorations parity

  • Portal build gate (2000🪙 + 5 runes → "Build Portal") and the summon economy are ported; Decorations (cost chips + influence grants + built counts) likewise.
  • The magic-tool effects from summoning route through the tool-power system and are only partially wired — a known M8 follow-up, not a UI gap.

Cartography / World map P3

  • React renders a rich, illustrated parchment world map ("The Hearthwood") with many clustered zone nodes, a legend, and a per-zone detail panel with lore and a flavour quote.
  • Godot renders a clean but minimal 3-node map (Hearthwood Vale · Cracked Quarry · Saltspray Harbor) with dashed roads, a legend, and a travel list ("Defeat the boss to unlock").
  • Functionally the travel/expedition gating works. The richer map + extra zones are the parked multi-settlement content — flagged as P3, not a regression.

Game menu P2

  • Branding matches ("🔥 Hearthlands — A puzzle of seasons and stews"); the "More" section routes the secondary screens.
  • Missing: a Settings submenu, an About page, a Game Wiki link, a Show Tutorial button, and Go Fullscreen. Godot inlines a single "Sound: On" toggle and "New Game".
  • The "More" list is a catch-all for secondary views, which differs from React's organisation (Trophies + Wiki as first-class entries).

Tutorial P2

  • Same copy, same 6-step flow, Skip / Next, scrim + parchment card.
  • Missing the NPC avatar (React shows Wren's portrait) and the page-dot indicator (Godot uses plain "Step 1/6" text).

Boons P3

  • React has a full keeper-perk economy: spend Embers (Coexist path) or Core Ingots (Drive Out path) on permanent per-biome boons (Deer-Blessing, Hearth-Thrift, Iron Market, Drilled Corps…), reached from a "✨ Boons" button on the town map.
  • Godot ships none of it — no Boons button, no screen, no ember/ingot currencies. Documented as an intentional drop (the keeper economy is unported); listed P3 so the team can confirm it stays dropped or schedule it.

How this review was done

  • React (reference): driven live via the Vite dev server + Playwright, using the in-app visual-scenario bridge (?visual=<id>) to render each screen/state, plus live interaction (entered a farm run, simulated a chain) through window.__hearthVisual.
  • Godot (port): rendered live with the project's own capture tooling (tools/batch_capture.gd, qa_grant_capture.gd, and review-specific scripts that seed farm/mine/harbor/boss runs and granted economies), at 720×1280 portrait.
  • Functional health: Godot headless suites re-run green during the review (scene smoke + board + economy, 0 failures).
  • Both sides were captured fresh and montaged left-vs-right; nothing was taken on faith from the parity matrix.
Reproduce. React caps: node tools/_review_react.mjs <scenario…>. Godot caps: the godot/tools/_rev_*.gd scripts. Montages: python tools/_review_montage.py → godot-parity-review/img/.

Intentionally dropped / parked (not bugs)

For completeness — these React surfaces are absent by design and should be confirmed, not "fixed" blindly:

Calendar seasons (gameplay)

The original removed the calendar-season cycle in Phase 7; the port keeps a turn counter and a lighter season strip. Recent work restored season-weighted spawns + season names.

Boons / festivals / run-summary recap

Keeper-economy-coupled systems. Boons + festivals fully absent; run-summary reduced to a 1-line harvest modal (see task 10).

Multi-settlement world

Founding, per-zone inventory, the full 11-node map, mine/harbor as foundable settlements. Parked behind a large refactor.

Mobile store export

iOS/Android signing + store accounts — externally blocked, not a code gap.

What the port already does well

  • The core chain → collapse → refill loop, with selection pulse, fly-to-coins juice, chain stars and escalating long-chain feedback.
  • Mine and harbor boards render their distinct tile art (ore/coal/gem, fish/shell/pearl) — strong parity.
  • The spatial town map (layout, river, pan/zoom, build flow) — only the building art lets it down.
  • Achievements, Quests/Almanac, Chronicle, Charter, Castle, Daily rewards, Decorations, Portal, Inventory, and the Start-farming modal are all near pixel-parity.
  • A persistent bottom-nav shell, opaque full-brightness views, smooth scrolling, emoji-font fallback, and a desktop framing pass are all in place.
