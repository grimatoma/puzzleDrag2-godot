Godot God-Object Decomposition

Design & implementation plan for the two large refactors deferred from the Godot-port best-practices review: extracting the HUD out of Main.gd (Track D) and splitting GameState.gd into composed domain objects (Track E).

Author: code-quality review follow-up · Status: Shipped — Track D #1046, Track E (Phase 1) #1048 · 2026-06-08 · Godot 4.6.2 port under godot/

Track D · HUD Track E · GameState plan-first CI-verified

Overview

A best-practices review of the Godot port found the codebase healthy on Godot idioms (signals-up/calls-down, dependency injection, 100% function return types, disciplined tween cleanup) with the weaknesses concentrated in two god-object files. The mechanical fixes (base classes, signal names, screen typing, doc cleanups, hot-path typed collections) shipped in PRs #1036 and #1037. This document planned the two remaining structural refactors — both now shipped (Track D #1046; Track E Phase 1 #1048) and kept here as the design record.

TL;DR. Both refactors are behavior-preserving decompositions of working code. Neither adds a feature. The whole point is to make the two largest files maintainable — so the bar is "the diff changes structure, not behavior," enforced by the headless test suite + a frozen save-format golden.

Track D — HUD extraction

Main.gd is ~2950 lines; a large share is HUD: top-bar pills, the chain-progress bar, the stockpile panel, plus every _refresh_* and the _layout math. Move it into a dedicated Hud node, leaving Main as the orchestrator.

Track E — GameState split

GameState.gd is ~2250 lines accreting many subsystems. Split into composed domain objects (e.g. fish/expedition) while preserving the exact persisted save shape and every reader.

The binding constraint: CI-only verification

These plans are written to be executed in an environment without a local Godot binary available to the author at planning time. That shapes every decision below:

  • The headless test suite is the compiler and the gate. godot-ci.yml runs --import (parses every script + builds the global class registry) then all 47 run_*.gd suites + the GdUnit4 suites, on every PR. A parse error or a broken contract fails it. The critical job ("Headless unit + scene tests") completes in ~50 s, so it is a fast, trustworthy gate.
  • No interactive / visual check. There is no way to look at the running game from the executing environment. Track D is visually sensitive, so its plan leans hard on "produce a byte-identical node tree" rather than "it looks the same."
  • No committed Linux visual goldens. The godot-visual job is render-smoke only (asserts a non-blank 720×1280 frame) and is continue-on-error; the pixel goldens live on the maintainer's machine. So CI cannot catch a visual regression — only a crash/blank. The maintainer must run test:visual-equivalent locally after Track D.
  • Pre-existing red: the godot-web-export job already fails on main (a Godot 4.6.2 headless-export shutdown SIGSEGV, unrelated to app code) — it is not a regression signal for these refactors and is tracked separately.
What CI can and cannot prove. Green CI proves the project still parses, all logic/scene tests pass, and the web build still boots. It does not prove the HUD still looks right (Track D) or that an old on-disk save still loads byte-for-byte (Track E) unless a test asserts it. Both plans therefore add an explicit guard for their blind spot.

Track D — Extract the HUD from Main.gd

Roughly 40–45% of Main.gd is HUD code: 38 member vars (top-bar pills, the chain-progress bar, the stockpile panel, the tool palette + armed banner, the bottom-nav, two FX/nav CanvasLayers) and 38 methods (7 _build_*, ~9 style/helper, 1 _layout_hud, ~21 _refresh_* / FX). The rest of Main.gd — Board ownership, the GameState economy bridge, ViewRouter, the lazy screens, audio, input, and the web History bridge — is the genuine orchestrator role and stays.

Goal. Move the HUD's nodes + build/refresh/layout/FX into a dedicated Hud (a CanvasLayer scene Main owns), so Main.gd drops to an orchestrator that says "something changed, refresh the HUD" rather than owning 38 widgets. Behavior-preserving: the node tree, styles, and layout math are moved verbatim, not redesigned.

The contract it must preserve

This is the crux. Main.gd's header comment already says it deliberately keeps the *_label fields and every _refresh_* method so "the scene-smoke assertion and the capture scripts that call the refreshers all keep working." The research confirms the exact surface that must remain reachable as main.X:

Fields (4)

main._chain_label · main._status_label · main._orders_label · main._tool_palette_box

run_scene_smoke.gd asserts _chain_label != null; run_tool_board_tests.gd reads _tool_palette_box; ~20 capture tools set _status_label.text.

Methods (11)

All _refresh_*: _totals · _meta · _settlement · _buildings · _orders · _biome · _boss · _rats · _runes · _chain_progress · _tools

Called from ~20 tools/*_capture.gd scripts to force a redraw before screenshotting.

Hard constraint. 30+ external call sites across godot/tools/*_capture.gd and 2 test files reference these 15 symbols as main.X. The extraction must not rewrite those call sites — it keeps the symbols on Main as forwarders to the new Hud.

Full external-reference list (30+ sites, from the codebase map)

Tests: run_scene_smoke.gd → _chain_label; run_tool_board_tests.gd → _tool_palette_box.

Capture tools (each calls a subset of the 11 refreshers + reads _status_label/_chain_label): action_capture, fish_capture, icon_capture, m3a–m3i_capture, m4a, m4b, m4e, m4f, m4g, m8d, toolbar_capture. m8d_capture + toolbar_capture additionally use _refresh_tools and _tool_palette_box.

Strategy: move implementation, keep the façade

The Hud node owns the 38 members and the build/refresh/layout/FX methods. Main keeps the 15 contract symbols as thin forwarders, so every external caller is untouched:

# Main.gd — after extraction
var _hud: HudScript                       # the extracted CanvasLayer, built in _ready

# Field forwarders (read-through; null-safe before _hud exists)
var _chain_label: Label:
    get: return _hud._chain_label if _hud else null
var _status_label: Label:
    get: return _hud._status_label if _hud else null
var _tool_palette_box: PanelContainer:
    get: return _hud._tool_palette_box if _hud else null

# Method forwarders (delegate the 11 refreshers)
func _refresh_totals() -> void: if _hud: _hud._refresh_totals()
func _refresh_meta()   -> void: if _hud: _hud._refresh_meta()
# … _settlement, _buildings, _orders, _biome, _boss, _rats, _runes, _chain_progress, _tools

The Hud needs read access to game (it reads game.inventory, game.coins, tier, biome, boss, etc.) and a few callbacks for nav taps. Inject those exactly like the other screens already do — var game: GameState set by Main, and signals up for taps (nav_selected(key), tool_armed(id), disarm_pressed) so the Hud stays unaware of routing. This matches the repo's established "signals up, calls down, DI not tree-reaching" pattern.

Why this order is safe. Commit 2 is a pure move + façade; a green headless suite after it is strong evidence the move was faithful (every _refresh_* still resolves, every test/tool symbol still exists). Blind spot. CI keeps no committed Linux visual goldens (render-smoke only). A HUD layout regression would pass CI. Track D therefore cannot be considered done on green CI alone — it needs the maintainer's local visual-golden run as a required step.

Track E — Split GameState.gd into domain objects

GameState.gd (~2250 lines) has accreted ~20 subsystems milestone-by-milestone — economy core, settlement, spawner buildings, refining, orders + NPC bonding, market, mine + harbor expeditions, boss gate, rats, tools, achievements, quests + almanac, story, workers, daily streak, settings, tutorial, castle, decorations, portal. The goal is to push each into a composed domain object (the way settlement: Settlement and story: StoryState already work), shrinking GameState to a thin aggregate.

The pattern already exists. Settlement and StoryState are already separate classes with their own to_dict()/from_dict(), held as fields on GameState and flattened into the save. They are the template for every other extraction — this track generalizes an existing, proven seam rather than inventing one.

The binding invariant: the save shape cannot change

Persistence is the dangerous surface. The port has no SAVE_SCHEMA_VERSION — it uses an additive-guard design: every field added since launch carries a defensive default in from_dict, so old saves load without migration. to_dict() flattens everything into a single-level Dictionary with 40+ keys; from_dict() defensively restores (type-coerces JSON floats→int, validates ids against Config, clamps ranges, drops malformed rows, auto-repairs incoherent states).

Invariant. A split must keep the persisted dictionary byte-for-byte identical. Sub-objects get their own to_dict(), but the parent GameState.to_dict() flattens them back into the same flat 40+ key shape; from_dict() builds the sub-objects from that flat dict (exact reverse). The on-disk format never changes — only the in-memory architecture does.

# GameState.to_dict() after a split — STILL emits the identical flat shape
func to_dict() -> Dictionary:
    var eco := economy.to_dict()        # { inventory, progress, coins, turn }
    var ord := orders_state.to_dict()   # { orders, npcs }
    return {
        "inventory": eco["inventory"], "progress": eco["progress"],
        "coins": eco["coins"], "turn": eco["turn"],
        "orders": ord["orders"], "npcs": ord["npcs"],
        # … every other key flattened from its sub-object, same keys as today
    }
# from_dict() builds economy/orders_state/… FROM the same flat dict (exact reverse).

Three load-time facts the split must not disturb: (1) pending_tool is transient — never serialized, always loads disarmed. (2) almanac_level is never trusted from the save — always recomputed from almanac_xp. (3) achievements_unlocked is restored so bump_counter stays idempotent (no double-grant on load). Each must live in exactly the sub-object that owns it, with the same handling.

The cross-cutting glue (why some seams are deferred)

A few responsibilities are deliberately woven through the economy and resist a clean cut. They are the reason the split is phased, not wholesale:

  • settlement.cap() gates every inventory write (credit_chain, craft, buy, story/quest grants) — extracting EconomyState away from Settlement risks splitting the single source of truth for storage limits.
  • post_story_event fires after credit_chain / try_tier_up / damage_boss / build / fill_order and reads inventory + settlement.tier for beat triggers. Story can't be lifted out without that context.
  • _tick_quests and bump_counter are called from the same event sites and mutate coins/tools while reading inventory — the quest/achievement layers are glue binding economy → progression.
  • active_biome drives active_tile_pool()/active_biome_pool() and gates rats/hazards — the expedition state must stay in sync with it.
Consequence. Extract the leaf subsystems (NPCs, tools, achievements, the small structs) first to prove the pattern + the save-shape preservation. Leave economy, story, and quests as the last-mile work — or wired as-is indefinitely. A smaller, cleaner GameState that still owns the glue is a perfectly good landing spot.

Recommended phasing

Phase 0 · confirm template

Document the Settlement/StoryState compose-into-flat-dict pattern as the canonical recipe. Add the save round-trip golden test (below) before touching anything — it must pass on today's code first.

Phase 1 · clean pilots

NpcState, then ToolState, then AchievementState. Each is one PR: create the class with to_dict/from_dict, move its fields + methods, flatten back in the parent, update readers (mostly game.npc_bond(...)-style calls stay as thin GameState forwarders so the ~8 reader files don't churn).

Phase 2 · medium

ExpeditionState (mine + harbor + tide + pearl), WorkerState. Verify Board/CartographyScreen flag-sync still holds.

Phase 3 · defer / optional

BossState, EconomyState, QuestState, lifting StoryState's wiring. Only if the glue is first untangled; otherwise stop after Phase 2.

Each domain object follows the same recipe, so prefer keeping GameState method names as forwarders (func npc_bond(id): return npcs.bond(id)) over rewriting the dozens of reader sites — the reader load is heavy (inventory 59 files, coins 44, settlement 30, story/quests/active_biome 22 each), and forwarders keep each PR small and its blast radius contained.

Verification guard: a save round-trip golden

CI's logic suites exercise the subsystems, but nothing today asserts the persisted shape is stable across a refactor. Before Phase 1, add a test that locks it:

# tests/run_save_roundtrip_tests.gd  (add to the run_*.gd suite)
# 1. Build a representative GameState (mid-run: coins, inventory, a placed building,
#    an order, an active expedition, a boss, some achievements, a quest, story progress).
# 2. var a := gs.to_dict()
# 3. var gs2 := GameState.from_dict(a)
# 4. var b := gs2.to_dict()
# 5. assert deep_equal(a, b)                      # round-trip is a fixed point
# 6. Also load a HAND-WRITTEN "legacy" flat dict (pre-NPC-layer order, missing the
#    additive keys) and assert from_dict fills the additive-guard defaults — proving
#    old saves still load after the split.
Why this is the right guard. It pins the exact contract a domain split is most likely to break — the flat key set and the additive-guard defaults — and it runs in the same fast headless suite as everything else, so every phase re-checks it for free. Land it green on today's code first; then any phase that perturbs the save shape fails immediately.

Sequencing

Do D before E. They are independent (HUD presentation vs. run-economy state), but Track D is the lower-risk of the two — it moves presentation code with a well-defined external contract (a handful of main._*_label reads + _refresh_* calls), and its blind spot (visual) is catchable by the maintainer with one local test:visual run. Track E touches persistence, whose blind spot (an old save failing to load) is subtler. Landing D first also shrinks Main.gd, making the codebase easier to reason about before the deeper E surgery.

Each track is its own PR with its own merge commit (matching the repo's fork-and-rejoin history convention), and each is internally phased so a reviewer can read it commit-by-commit. Within a track, split the mechanical "move" from any "rewire" so a regression bisects to a small commit.

Verification strategy

Every phase ends green on the same gate the shipped tracks used. The two refactors each add one guard for their specific blind spot, because CI alone cannot see it.

Shared principles

Preserve contracts, move implementations

Both god-objects have an external contract (tests, tools, savefiles, readers). The decomposition moves the implementation behind that contract; where the contract names a symbol on Main/GameState, keep a thin forwarder/property rather than rewriting every call site.

Mechanical move ≠ behavior change

Split each PR so the pure "cut/paste into a new unit + wire it up" commit is separate from any logic touch. A green test after the move commit is the strongest possible signal the move was faithful.

No new features, no balance touches

These are structure-only. Resist the urge to "fix" adjacent code — it muddies the diff and the regression signal.

One subsystem at a time (E)

Extract the lowest-coupling domain object first as a pilot to prove the pattern + the save-shape preservation, then repeat. Never split everything in one commit.
