Godot Migration Plan

Strategic rationale and phased roadmap for migrating puzzleDrag2 from React + Phaser 3 to Godot 4. Phase 1 (M1–M4) shipped a playable Town 1→2→3 vertical slice. Phase 2 (M5–M11) pursues full feature parity — logic-first, mobile-first, with GdUnit4 + e2e + visual golden-diff.

Status: Finished slice · 2026-06-06 · Phase 1 done · Phase 2 done (parity 54% over in-scope rows) · residual is parked / externally-blocked / polish

Godot 4 GDScript Mobile App CI/CD Asset Pipeline

Decision

TL;DR. Migrate to Godot 4 now, running both versions side-by-side in one repo (godot/ subdirectory). Do not implement the remaining gameplay mechanics (demolish, tier-up plots, expeditions, capstone boss) in the Phaser codebase — build them directly in Godot against the locked Direction spec. Delete the Phaser version once Godot reaches M3 feature parity.

The primary target for this game is a mobile app (iOS + Android). That single fact makes Godot the right choice: it exports natively to both platforms with real OpenGL/Vulkan rendering, while a React + Phaser app wrapped in Capacitor hits a WebView performance ceiling and carries App Store approval risk.

The Direction wiki page (PR #923) has landed a clear, locked spec for Town 1 → Town 2 (capstone boss) → Town 3. With that spec in place, there is no design uncertainty that requires the Phaser prototype to keep running. Building the remaining mechanics in Phaser and then reimplementing them in Godot pays the cost twice.

Why Godot, why now

Mobile performance

Godot exports native iOS .ipa and Android .apk/.aab. Real OpenGL/Vulkan — no WebView ceiling. Mid-range Android devices that stutter on canvas-in-WebView run Godot games without issue.

App Store approval

Apple increasingly rejects apps that are "just a website in a WebView." A native Godot build has no such flag. This is a distribution risk with the Capacitor-wrapped Phaser approach.

Game-first tooling

Tilemaps, animation state machines, physics, scene trees, and built-in touch input are first-class in Godot. The current React/Phaser bridge (phaserBridge.js, the registry hand-off, dual state ownership) goes away entirely.

Don't build twice

Any mechanic (demolish, tier-up plots, expedition flow, capstone boss) built in Phaser now gets thrown away. The Direction spec is the implementation target — build against it once, in Godot.

Migration strategy — side-by-side in one repo

The Godot project lives in a godot/ subdirectory of the existing repo. The Phaser version stays at the root and remains fully functional throughout the migration. Both are committed to main; neither is on a long-lived branch.

Why one repo. The Dev Panel, Story Editor, Wiki, balance constants, story content, and all docs are shared by both versions. Keeping them in one repo means there is one source of truth for design and data — no syncing between repos, no duplication.

Re-baseline — 2026-06

Phase 1 shipped. The original M1–M4 arc delivered a playable Town 1 → Town 2 → Town 3 vertical slice: board logic, core game systems, content & progression, and a first-pass UI layer. The current game textures are Canvas 2D procedural drawings generated in JavaScript at Phaser scene init (src/textures/categories/). They are not SVG files — they're rasterized at runtime into Phaser's texture cache and cannot be transferred to Godot directly. The path forward is a three-stage progression from placeholders to exported v1 PNGs to full v2 animated sprite sheets.

Stage 1 — Colored placeholders (M1)

Do not use real tile art during Milestone 1. The milestone proves the mechanic works, not the art. In Godot each tile is a ColorRect or a Sprite2D with a 1×1 pixel solid color texture — one color per tile type.

# tile.gd — v1 placeholder color lookup
const TILE_COLORS = {
    TileKey.GRASS:  Color(0.42, 0.68, 0.32),
    TileKey.STONE:  Color(0.55, 0.55, 0.55),
    TileKey.WATER:  Color(0.28, 0.52, 0.82),
    # … one entry per TileKey enum value
}

Zero asset cost. Replace with real PNGs in Stage 2 by swapping the texture on the same Sprite2D node — the surrounding code does not change.

Stage 2 — Export v1 PNGs once (M2 / M3)

Build a one-time Playwright export script that drives the existing Phaser web app headlessly, extracts every texture from Phaser's cache, and saves each as a named PNG. The current game already exposes window.__phaserScene in dev mode, making this straightforward.

Export script outline

// tools/export-v1-tiles.mjs
// Run once: node tools/export-v1-tiles.mjs
// Writes assets/tiles/*.png ready to drop into res://assets/tiles/

import { chromium } from 'playwright';
import { writeFileSync, mkdirSync } from 'fs';

const browser = await chromium.launch();
const page    = await browser.newPage();
await page.goto('http://localhost:5173/puzzleDrag2/?visual=board-farm');
await page.waitForFunction(() => window.__phaserScene?.textures);

const pngs = await page.evaluate(() => {
  const tm  = window.__phaserScene.textures;
  const out = {};
  for (const key of Object.keys(tm.list)) {
    if (key === '__DEFAULT' || key === '__MISSING') continue;
    const src = tm.get(key).getSourceImage();       // HTMLCanvasElement or HTMLImageElement
    const c   = document.createElement('canvas');
    c.width   = src.width;  c.height = src.height;
    c.getContext('2d').drawImage(src, 0, 0);
    out[key]  = c.toDataURL('image/png');
  }
  return out;
});

mkdirSync('assets/tiles', { recursive: true });
for (const [key, dataUrl] of Object.entries(pngs)) {
  const buf = Buffer.from(dataUrl.split(',')[1], 'base64');
  writeFileSync(`assets/tiles/${key}.png`, buf);
  console.log(`  wrote ${key}.png  (${buf.length} bytes)`);
}
await browser.close();
Run once, commit the PNGs. The output folder (assets/tiles/) gets committed to the Godot repo. This is a bootstrap — you run the script once to seed the folder, then only re-run it if a v1 texture is intentionally changed in the Phaser codebase before v2 replaces it.

Using the PNGs in Godot

# tile_visual.gd
func set_tile(key: TileKey) -> void:
    var path := "res://assets/tiles/%s.png" % TileKey.keys()[key].to_lower()
    texture = load(path)

Godot caches load() calls — no performance concern for a 6×6 board.

Stage 3 — v2 animated sprite sheets (M4+)

As each v2 asset arrives from PixelLab / Ludo.ai, it replaces the corresponding v1 PNG. In Godot the swap is a node-level change — replace Sprite2D with AnimatedSprite2D and point it at a SpriteFrames resource. The rest of the tile code is unchanged.

Sprite sheet format for Godot. Export from PixelLab / Ludo.ai as a horizontal strip PNG (all frames in one row) or as individual frame PNGs. Godot's SpriteFrames editor imports both. Prefer transparent background PNGs at the tile's native resolution (32×32 or 64×64 to match the board grid). Avoid atlases with mixed tile types in Stage 3 — keep one file per tile for clean per-tile swapping.

Visual consistency across v1 and v2

Because each tile's visual is fully encapsulated in its own Godot scene, v1 and v2 tiles can coexist on the same board during the transition — there is no "all-or-nothing" art swap. Grass goes v2 while stone stays v1, and they render side by side without any code changes.

The only consistency requirement is that all tile scenes expose the same interface to the board:

# Every tile visual scene (v1 or v2) implements this
func set_selected(on: bool) -> void: …   # highlight / deselect
func play_collect() -> void: …           # collect animation
func get_tile_key() -> TileKey: …

Godot MCP — local dev accelerator

Godot MCP servers (Godot MCP Pro, GDAI MCP, open-source alternatives) connect Claude Code to a live running Godot editor via an editor plugin. ~15 min setup. 163 tools covering scenes, nodes, scripts, animation, physics, and more.

This works on your local machine with the editor open. It does not replace CI — it accelerates local development (scene inspection, node creation, script attachment, running the project) in the same way the current Phaser MCP for Aseprite accelerated asset generation.

Recommended setup. Install a Godot MCP server locally alongside the Godot editor for interactive development. Use GdUnit4 headless + HTML5 export for CI and cloud container verification. The two workflows are complementary.

Pixel art asset pipeline

The procedural Canvas 2D textures in src/textures/ are a Phaser/browser-only pattern. In Godot, sprites are pre-generated PNGs loaded from res://assets/.

Recommended pipeline for generating assets:

  • PixelLab.ai — skeleton-based animation, 4/8-direction rotation, tileset generation. Best for character sprites and directional assets. API available for pipeline integration. ($12–50/mo)
  • Ludo.ai — duration-based animation (up to 4s), transparent PNG sprite sheet export, native MCP integration. Good for quickly animating existing sprites. Free tier (30 credits).

Both export transparent PNG sprite sheets sized for Godot's AnimatedSprite2D or Sprite2D nodes.
