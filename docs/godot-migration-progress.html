M3a landed the progression spine — the Camp→City tier ladder: a single town-center upgrade that costs escalating local resources and raises a storage cap, plot count, and the next batch of building unlocks. M3b added the central "the town you build reshapes the board" mechanic: every non-staple tile category enters the refill pool only because a spawner building (Lumber Camp, Coop, Garden) put it there, plots are scarce, and demolish frees a plot and drops its category. M3c now closes the loop with the refining + market economy: a Bakery refiner bakes raw goods into bread (3 flour + 1 eggs → 1 bread) via station-gated recipes, and a cap-aware Market sells goods for coins and buys them back. M3d adds the Orders system — the Direction spec's coin sink: an NPC requests a quantity of a resource you can produce, and filling the order pays 3× the raw Market price, so delivering goods beats selling them. M3e now ships the real Town UI screen — a Control-based modal (built in code, like the HUD) with five live sections (Settlement, Buildings, Refine, Market, Orders) whose buttons drive the whole Town-1 economy, replacing the temporary T/1-6/B/G/F dev keys (which remain only as a keyboard fallback). M3f opened the mine expedition (the food gate → Town 2), M3g landed the Direction's dramatic ★ capstone boss (Frostmaw — min_chain 4 while engaged, raising the bar to resolve a chain — closing Town 2), and M3h shipped Town 3's rats hazard + Ratcatcher (rats are a no-yield tile seeded into the farm pool once Town 2 is done; Ratcatcher shoos them, Master Ratcatcher folds 8-adjacent rats into a grass chain). Decisions & deviations from the plan

Each is a deliberate call made while building M1. Where it deviates from the strategy doc, the rationale is noted so a future reader (or a reviewer) can challenge it.

1 · Constants is a class_name global, not an autoload

The plan calls for a Constants autoload. Autoloads aren't guaranteed to be instantiated when a script is run via --script (headless tests), so a pure class_name with const/enum/static func members is reachable at compile time from tests with no live tree. This satisfies the plan's intent (one shared constants module) while keeping tests dependency-free. An autoload instance can still be added later if a singleton with mutable state is needed.

2 · Lightweight headless test runner now; GdUnit4 later

The plan specifies GdUnit4. Vendoring that addon needs a network fetch that could block "tests from day one," so M1 ships a tiny dependency-free runner (extends SceneTree, asserts, quit(0/1)) that CI gates on. It exercises the exact same BoardLogic class GdUnit4 will wrap, so adopting GdUnit4 is additive, not a rewrite. Tracked as a partial M1 item.

3 · Web export uses the nothreads variant

variant/thread_support=false. Threaded WebAssembly requires the host to send COOP/COEP cross-origin-isolation headers; GitHub Pages does not. Single-threaded keeps the export runnable on Pages (where the dev tools already deploy).

4 · Mechanic rules verified against source, not assumed

Adjacency is 8-way (king moves), matching is by exact tile key, the path may not revisit a cell, and dragging back onto the previous cell backtracks. All confirmed against src/GameScene.ts:1434–1436 before porting.

5 · Visual layer is authoritative; grid is re-synced from it

Board mutates its tile nodes during collapse/refill, then derives the integer grid from them (_sync_grid_from_tiles). This avoids keeping two sources of truth in lock-step and guarantees drag validation always sees exactly what's on screen.

6 · Minimal animation for M1 (by design)

Collected tiles pop, survivors slide down, new tiles fall in — enough to "feel right". Full collect/collapse VFX (particles, golden upgrade burst, staggered timing) is explicitly M4 and is not attempted here.

7 · Cross-collection deferred to M2/M3 — flagged so it's not lost

The Phaser game also collects tiles of a different family that are 4-way adjacent to the chain (src/GameScene.ts:1769–1772). M1 ports the single-family chain only. Cross-collection is a resource-systems detail and now rides along with the M3 content/biome work. Recorded here as an explicit debt.

M2 decisions

8 · The carry-over accumulator lives in GameState, not the board

In M1 the board computed units per-chain, but a per-chain calculation cannot carry a remainder across chains. M2 moves the whole resource credit into GameState.credit_chain and narrows the board signal to chain_resolved(tile_type, length) — the board reports the family + length and knows nothing about resources or economy. credit_chain ports the React resourceProgress accumulator from src/state.ts (the CHAIN_COLLECTED site) exactly: units = (progress[res] + n) / threshold, with progress[res] = (progress[res] + n) % threshold persisting the remainder between chains. This keeps a single owner of run state and makes the carry-over behaviour unit-testable in isolation.

9 · Save/load discards on mismatch — it does not migrate

SaveManager.load_state() discards a save that is missing, unparseable, or version-mismatched and starts a fresh run, rather than maintaining forward migrations. This is the deliberate policy the React app uses with SAVE_SCHEMA_VERSION (the reducer discards saves whose version doesn't match). Bumping the version is the sanctioned way to invalidate old saves; there is intentionally no migration ladder to maintain.

10 · No season-of-year calendar — a plain monotonic turn instead

The strategy doc's M2 list mentions a "10 turns × 4 seasons" cycle. But the live React app removed the season calendar from gameplay in Phase 7 — SEASONS is now visual-only and no longer drives harvest bonuses, order multipliers, or a winter min-chain floor. So M2 implements a plain monotonic turn counter and ports no season mechanics. This is faithful to the current code, not the stale plan.

11 · Coins simplified for M2 — full value economy is M3

Each resolved chain earns max(1, chain_len / 2) coins. The React per-tile value-based coin economy (and the XP/almanac, bubbles, and hazards that go with it) is deferred to M3. M2's coin counter exists to prove the HUD and persistence wiring end-to-end; the real numbers land with the content/progression milestone.

M3 decisions

12 · Storage cap enforced in credit_chain, not a separate pass

The Direction "cap" (200→600 by tier) is applied at the credit site: credit_chain clamps each resource to settlement.cap() as it writes it. This keeps cap enforcement simple and central — there is one place that mutates inventory and it is also the place that respects the ceiling, so no resource can slip past the cap through a side path. Revisit if per-resource or soft (overflow-with-decay) caps are needed later; for the first-pass single global cap this is the least machinery.

13 · Temporary T-key tier-up affordance until the M3e Town UI

The real tier-up button belongs to the Godot town screen, which is a later sub-PR (M3d). Rather than block the ladder on that UI, M3a drives the town-center upgrade with a keyboard T key so the whole Camp→City progression is exercisable and demoable today (and the 69-check suite can assert it). It is flagged temporary both in Main.gd and here, and is removed when M3d ships the button.

14 · Tier-up costs are PC2-aligned first-pass values (tunable)

The escalating, cross-category tier-up costs in TownConfig.gd are a first-pass aligned to the PC2 balance baseline and expressed only in resources the port already produces, so the ladder is playable today. They live as data in TownConfig and are tunable — same spirit as the M2 coins simplification (Decision #11); final balance rides with the broader M3 content work. Updated in M3b: once the board pool became building-gated, the M3a first-pass costs would deadlock (they required plank to reach Hamlet, but plank needs the Lumber Camp, which only unlocks at Hamlet), so the ladder was rewritten to be gating-aware — see Decision #16 for the new values.

M3b decisions

15 · The board's refill pool is building-gated and dynamic

Beyond a town's two staple tiles (grass→hay_bundle, wheat/grain→flour, weighted in STAPLE_POOL), every other tile category enters the spawn pool only because a spawner building put it there. GameState.active_tile_pool() = staples + each placed spawner's tile (twice), and it is swapped into the board via Board.set_tile_pool; setup, reshuffle, and refill all draw from that dynamic pool, so building a spawner makes its category start appearing as the board refills. This is the Direction's "the town you build reshapes the board you play" — the central M3 mechanic, expressed with the least machinery.

16 · Tier costs revised to be gating-aware (deadlock-free)

Once the pool is gated (Decision #15), M3a's costs would deadlock — the →Hamlet cost required plank, but plank only exists once the Lumber Camp is built, and the Lumber Camp only unlocks at Hamlet. So the Town-1 ladder costs were rewritten so every cost references only resources producible at or below the tier it gates: →Hamlet {hay_bundle:12, flour:6}, →Village {plank:8, hay_bundle:16, flour:8}, →Town {eggs:8, soup:6, plank:10}, →City {soup:10, eggs:12, plank:14}. The run_building_tests.gd no-deadlock progression walk proves the whole Camp→City climb is reachable by building each spawner and tiering up at each step.

17 · Demolish has no resource refund (first pass)

The Direction lists the refund policy — and whether a progression-required building should be non-demolishable — as open questions. M3b takes the simplest stance: demolish frees the building's plot and drops its category from the pool, but returns no resources. It is flagged temporary; the refund/lock policy is decided when the M3e Town UI makes demolish a deliberate player action rather than a dev key.

18 · Temporary 1/2/3/4/5/6 build/demolish keys until the M3e Town UI

The real build/demolish menu belongs to the Godot town screen (M3d). Rather than block the buildings mechanic on that UI, M3b drives it with keyboard keys — 1/2/3 build the three Town-1 spawners, 4/5/6 demolish them — so the whole build → re-pool → category-spawns loop is exercisable and demoable today (and the 90-check suite can assert it). Same spirit as the temporary T-key tier-up affordance (Decision #13); the keys are removed when M3d ships the menu.

M3c decisions

19 · Refining is station-gated recipes, deducting inputs for a cap-clamped output

The Direction's "raw goods refine at buildings" loop is modelled as recipes that name a station building. GameState.craft requires that station to be placed, deducts the recipe's exact inputs, and adds the (cap-clamped) output — guard order unknown → no_station → insufficient, never mutating on failure. The first recipe is bread (3 flour + 1 eggs → 1 bread) at the Bakery, ported from the React RECIPES (rec_bread). The catalog lives as data in RecipeConfig.gd and is the single place a station-gated transform is defined, so adding the Mill (grain→flour) or any later refiner is a data row, not new control flow.

20 · Market buy is cap-aware; sell credits sell_price × qty

Selling is straightforward — sell credits coins += sell_price × qty (guard order bad_qty → not_sellable → insufficient). Buying is the subtle one: it respects the settlement storage cap, so buy only purchases and charges for the units that actually fit — cant_afford when you can't pay the full ask, cap_full when storage is already maxed, and a partial fill when only some units fit. This keeps the cap (Decision #12) authoritative on the spend side too: coins can never conjure inventory past the ceiling. The SELL/BUY tables are PC2-aligned first-pass values in MarketConfig.gd (tunable), same spirit as the tier-cost first pass (Decision #14).

21 · Buildings carry a kind — spawners seed the board, refiners do not

Adding the Bakery surfaced a split in what a building does: the three category buildings are spawners (they put a tile category into the board pool — Decision #15), but the Bakery is a refiner (it enables a recipe and adds nothing to the pool). So BuildingConfig buildings now carry a kind ("spawner" | "refiner"), and active_tile_pool()/active_categories() guard on is_spawner so a placed Bakery never seeds a blank tile/category onto the board. Same building list, two roles — the dynamic pool stays exactly the placed spawners.

M3d decisions

22 · Orders pay 3× the Market price

Filling an order beats selling the same goods at the Market: an order's reward is sell_price × qty × 3 (REWARD_MULT = 3 in OrderConfig.gd), so delivering goods to an order is the preferred coin sink — exactly the Direction spec's intent. The reward is uncapped: only inventory resources hit the settlement storage cap (Decision #12); coins, like sell income, do not. The multiplier and qty range are PC2-aligned first-pass values living as data in OrderConfig, tunable in the same spirit as the tier-cost and Market first passes (Decisions #14, #20).

23 · Orders are always fillable-in-principle

A generated order only ever requests a resource in orderable_resources() — the two staples plus each placed building's resource — so the player can never be handed an order for a good they have no way to produce. generate_order draws from that set (seeded by the GameState rng for reproducible tests), and refill_orders tops the board up to MAX_ORDERS idempotently. The orders suite asserts this invariant across ~20 generated orders.

M3e decisions

24 · The Town UI is built in code as a CanvasLayer modal

Like the HUD, the town panel is constructed in GDScript with no .tscn authoring — a dim backdrop over a scrollable PanelContainer with five live sections (Settlement, Buildings, Refine, Market, Orders). Its buttons drive the existing GameState methods (tier-up / build / demolish / craft / sell / fill), and the panel re-renders after each action so the enabled/disabled state of every button stays accurate against the current state. This makes the panel the real path for the Town-1 economy and demotes the temporary T/1-6/B/G/F dev keys (Decisions #13, #18) to a keyboard fallback. Building the modal in code keeps the side-by-side port consistent with the HUD and lets the 35-check suite press any specific button via the exposed _action_buttons map.

25 · Safe refresh() under signal emit — detach now, free deferred

A button's pressed handler mutates GameState and then re-renders, which rebuilds the rows that contain the very button still emitting its signal. So refresh() detaches the old rows immediately (remove_child) but defers the node free (queue_free) — a button is never freed under itself while a signal is being emitted, and the _action_buttons lookup stays consistent within the same call stack. This is the fix for the "freed while a signal is being emitted" class of bug; detaching synchronously keeps the live tree correct, deferring the free keeps the in-flight emit safe.

M3g decisions

26 · The boss raises the chain floor to resolve, not the dead-board check

An engaged capstone boss (Frostmaw) raises the board's minimum chain to min_chain 4 — but only for resolving a chain. Board.gd carries a dynamic min_chain that is passed to both is_valid_chain resolve sites, so while the boss is alive the player must drag a longer chain to land a hit. The dead-board / has_valid_chain check deliberately stays at the base minimum (3): it answers "does a playable move exist at all?", and tying it to the raised boss bar would falsely declare a board dead (and trigger an auto-reshuffle) whenever only 3-chains remained. Two distinct questions — "is this resolution legal?" (raised) vs "is the board still alive?" (base) — so two distinct thresholds. This ports the React boss.minChain behaviour faithfully.

M3h decisions

27 · Rats are a no-yield hazard tile, gated on town2_complete

Town 3's fixed lesson is a hazard, not a resource. Constants.Tile.RAT (ordinal 15, append-only so existing save enum ints stay stable — Decision #9 / the save-schema risk) produces nothing and has no threshold, so chaining rats only wastes a move. They appear only after Town 2 is cleared: rats_enabled() = town2_complete seeds RAT_POOL_SLOTS=2 rats into the farm active_tile_pool() (the mine pool is left untouched, since the hazard is a Town-3 farm lesson). Gating the hazard on the boss-clear flag keeps the introductory difficulty curve in order — rats can't surprise a Town-1 player.

28 · Ratcatcher charges vs Master-Ratcatcher adjacency clear — two distinct counters

The hazard has two answers, modelled as a new building kind: "hazard" (City unlock, gated on rats_enabled). The Ratcatcher is an active, metered tool: 5 shoo charges (ratcatcher_charges_left / can_shoo_rats / use_ratcatcher_charge) drive Board.clear_all_rats(), a free move that blanks every rat then collapses + refills. The Master Ratcatcher is passive geometry: Board.clear_rats_on_grass folds any rat 8-adjacent to a resolving grass chain into the same removal set — the rats are cleared as a side-effect of normal play, but are deliberately not counted or credited (they yield nothing — Decision #27). Separating the metered charge from the free adjacency sweep keeps each upgrade's value legible.

M3i decisions

34 · The mine hazard reuses the rats pattern, generalized — biome-distinct

The mine hazard (rubble / cave-ins) is the rats hazard pattern generalized: a no-yield board tile that a player clears by chaining a trigger tile adjacent to it. Constants.Tile.RUBBLE (ordinal 16, append-only so existing save enum ints stay stable — Decision #9 / the save-schema risk) produces nothing and has no threshold, so chaining rubble only wastes a move — exactly the rats invariant (Decision #27). But it is deliberately biome-distinct from rats on every axis: it seeds into the mine pool (not the farm — active_biome_pool() seeds RUBBLE_POOL_SLOTS=2 rubble only inside an expedition, leaving the farm pool untouched), it is cleared by a resolving STONE chain (not grass — Board.clear_rubble_on_stone folds 8-adjacent rubble into the removal set, uncredited, so you "mine through it"), and it is built-in rather than building-gated (there is no Ratcatcher equivalent — the urgency comes free from the limited mine turns / the supplies-food gate). Reusing the proven adjacency-clear shape kept the hazard cheap to build and legible to a player who already learned the rats lesson, while the per-axis differences keep the mine feeling like its own place. Rubble is board-only (no save-shape change), like rats.

M3f / scope decisions

29 · One shared inventory for the whole slice — per-settlement split deferred

The Direction ultimately wants per-settlement resource pools (each town hoards its own goods). The M3a–M3h slice deliberately uses one shared GameState inventory across Town 1, the mine expedition, and Town 3, so the vertical slice could be built and verified end-to-end without the bookkeeping of a multi-pool model. Recorded here as an explicit, honest debt — alongside the other deliberately deferred pieces: the Town-2 tier ladder and Town-3's broader specialization roster (the Direction's draft beyond the fixed rats lesson). These land with the broader M4 content/parity work, not the introductory slice. (The mine hazard deferred here has since been built — see M3i / Decision #34.)

M4 decisions

30 · M4 ports the original's parchment palette/aesthetic, with board chrome drawn in code

M4 replicates the original React+Phaser game's cozy "leather-bound journal" look rather than inventing a new one: the palette in Palette.gd is ported directly from the original's src/tokens.css tokens (parchment surfaces, ink text, iron border, ember/gold/moss accents, field tint, chain-path colors), so the Godot build reads as the same game, not a re-theme. The board chrome is drawn in code — the field-tinted board card is a StyleBoxFlat rendered in Board.gd's _draw(), and the chain-path overlay is a Node2D draw in ChainOverlay.gd — with no .tscn art assets, keeping the visual layer consistent with the already-code-built HUD and Town modal (Decision #24) and free of binary scene files to review.

31 · The HUD top-bar shows the port's real state — no level / XP / season pills

M4b mirrors the original's top-bar structure (a title + right-aligned status pills) but only surfaces state the Godot port actually has: coins, tier·plots, and biome, plus a boss pill and a rats pill that appear only while those systems are live. The original's level / XP and season pills are deliberately absent — the season calendar was removed from gameplay (Decision #10) and the per-tile value economy / XP-almanac layer is deferred (Decision #11), so showing those pills would be decoration with nothing behind it. The HUD is an honest readout of the slice that exists, not a pixel-copy of the original's chrome; the missing pills come back if and when their systems are ported.

32 · Audio is synthesized at runtime; the Audio node is owned by Main, not an autoload

M4d ports the original's WebAudio SFX without shipping any audio asset files: Audio.gd synthesizes each of the 9 sounds into an AudioStreamWAV at init, mirroring the original's oscillators (src/audio/index.ts: sine/square/triangle/sawtooth + frequency glide + an attack/decay envelope) — so the port stays asset-free and the sounds are tunable as code, the same posture as the procedurally-drawn board chrome (Decision #30). And the Audio node is owned by Main rather than registered as an autoload, so the headless --script test harness stays autoload-free (consistent with Decision #1) — the logic suites never depend on a live audio singleton, and the 63-check run_audio_tests exercises the service directly.

33 · Board animations are tween-driven in code; reward chips live on a dedicated FX layer

M4e ports the original's board juice (the rewardTrajectory reward-fly + tile spawn pop) with no AnimationPlayer and no .tscn art — every effect is a code-driven Tween on a node built at runtime, consistent with the procedurally-drawn board chrome (Decision #30) and the asset-free audio (Decision #32). The reward chips render on a dedicated _fx_layer (a CanvasLayer at layer 2) above the HUD, so a chip flying to the coin pill is never clipped by or tangled with the HUD's own layout, and the FX are cleanly separable from gameplay nodes. The pass is purely cosmetic — no economy/board-logic change — so all suites stay green with no test edits.
